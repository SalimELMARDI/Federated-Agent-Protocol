# Governed Execution Flow

This document describes the currently implemented `participant_docs /execute` path and the matching coordinator-managed execution dispatch.

## Participant Endpoint

- `POST /execute`

## Expected Inbound Message

Currently implemented input:

- canonical FAP JSON object with top-level `envelope` and `payload`
- parsed through the shared parser
- only `fap.task.create` is supported for execution

If the inbound valid FAP message is not `fap.task.create`, `participant_docs` returns:

```json
{
  "detail": {
    "code": "unsupported_execution_message",
    "message": "participant_docs can only execute 'fap.task.create' messages"
  }
}
```

Malformed known-kind messages use the shared parse error mapping and currently return HTTP `422` with `code = "invalid_message"`.

## Deterministic Local Docs Search

`participant_docs` currently uses a hardcoded local dataset of four docs. Search behavior is:

- case-insensitive
- match if the query appears in doc title or content
- stable match order based on the dataset order
- empty result if no docs match

## Local Content Construction

Currently implemented local content rules:

- if one or more docs match:
  - local content becomes `Matched docs: <title1>; <title2>; ...`
- if no docs match:
  - local content becomes `No relevant docs found for query: <query>`

## Governance Defaults

Before policy is applied, `participant_docs` derives governance inputs from the inbound envelope.

Current defaults when `envelope.governance` is missing or incomplete:

- `privacy_class = internal`
- `sharing_mode = summary_only`
- `policy_ref = "policy.docs.v0"`

## Policy Engine Application

`participant_docs` constructs a shared `LocalResult` and calls the shared policy engine. It does not implement policy transformation rules locally.

## Returned Message Pair

`participant_docs /execute` currently returns HTTP `200` with:

```json
{
  "task_complete": { "...": "..." },
  "policy_attest": { "...": "..." }
}
```

### `task_complete`

Current behavior:

- message type is `fap.task.complete`
- preserves inbound `task_id`, `run_id`, and `trace_id`
- `sender_id = "participant_docs"`
- `recipient_id = inbound sender_id`
- `domain_id = "participant_docs"`
- `summary` uses the approved export content
- if the approved export content is `null`, summary becomes `[VOTE ONLY] No content exported`

### `policy_attest`

Current behavior:

- message type is `fap.policy.attest`
- generated by the shared policy engine
- uses preserved run context supplied by `participant_docs`
- `sender_id = "participant_docs"`
- `recipient_id = inbound sender_id`
- `domain_id = "participant_docs"`

## Provenance Correlation

When run context is supplied, the current implementation preserves:

- inbound `task_id`
- inbound `run_id`
- inbound `trace_id`

This makes the returned `policy_attest` part of the same FAP run as the related `task_complete`.

## Coordinator-Managed Execute Dispatch

The coordinator currently exposes:

- `POST /runs/{run_id}/dispatch/participant-docs/execute`

Current high-level behavior:

- load the stored original `fap.task.create`
- send it to `participant_docs /execute`
- require both `task_complete` and `policy_attest`
- parse both through shared FAP parsing
- record both into the in-memory coordinator run state
- return the canonical execution payload to the caller

Current coordinator error mapping:

- unknown run -> `404 run_not_found`
- non-200 downstream execution -> `502 participant_execution_failed`
- malformed or invalid execution payload -> `502 invalid_participant_execution_response`

# FAP v0.2 Plan

## Objective

Move FAP from a strong protocol alpha with reference participants to a more concrete system for
coordinating discoverable domain agents across trust boundaries.

The v0.2 goal is not to become a full decentralized agent network. The goal is to make the current
coordinator-managed model:

- more routable
- more self-describing
- more useful for real multi-domain workflows

## Core Architectural Principle

The architectural line for v0.2 should be:

- **FAP between participants**
- **MCP inside participants**

This means:

- FAP remains the inter-domain coordination and governance protocol
- MCP remains a participant-local tool/runtime interface
- a participant should increasingly look like a domain agent runtime, not only a connector

## v0.2 Goals

### 1. Make discovery operational

Participant profile and status should become routing inputs, not only metadata.

### 2. Introduce routing constraints

Task creation should be expressive enough for the coordinator to choose participants intentionally.

### 3. Support richer evidence semantics

The protocol should better distinguish between evidence and provenance kinds.

### 4. Add progress and follow-up support

Participants should be able to report intermediate progress, and the coordinator should be able to
support follow-up orchestration without inventing a separate side channel.

### 5. Clarify participant classes

The protocol should explicitly distinguish:

- local participants
- outbound participants
- hybrid participants

## Non-Goals For v0.2

The following should not be the main focus of v0.2:

- full peer-to-peer A2A
- decentralized participant-to-participant topologies
- broad transport proliferation before semantics are stable
- production-grade enterprise trust infrastructure in one jump

## Protocol Changes

### 1. Extend `fap.task.create` with routing constraints

Recommended additions:

- `required_capabilities`
- `privacy_requirement`
- `max_latency_ms`
- `cost_budget`
- `allow_outbound_model_calls`
- `preferred_participants`
- `fallback_allowed`

Expected outcome:

- the coordinator can reason about who should be involved
- outbound participants like `participant_llm` can be excluded unless explicitly allowed

### 2. Add `fap.task.progress`

Purpose:

- allow a participant to report meaningful intermediate progress
- improve transparency for longer-running or more agentic flows

Suggested payload fields:

- `participant_id`
- `stage`
- `progress_note`
- `percent_complete`
- `partial_source_refs`

### 3. Upgrade evidence/provenance typing

The current `source_refs` model is useful but too generic.

Recommended additions:

- `evidence_kind`
  - `document`
  - `log`
  - `kb`
  - `tool_output`
  - `model_output`
  - `external_provider`
- `provenance_kind`
  - `local_source`
  - `derived`
  - `external_service`
- optional `confidence`
- optional `citation_text`

### 4. Clarify participant profile fields as operational inputs

The profile already carries:

- `execution_class`
- `latency_class`
- `cost_class`
- `supports_mcp`
- `supports_followup`
- `outbound_network_access`

In v0.2 these fields should actively affect routing decisions.

## Runtime Changes

### 1. Make coordinator routing discovery-driven

The coordinator should:

- fetch profile/status discovery for trusted participants
- filter out unhealthy participants
- filter by requested or required capabilities
- enforce privacy and outbound-call constraints
- select participants intentionally instead of only dispatching by static path

### 2. Keep `participant_llm` in a coherent participant model

Immediate follow-up:

- keep `participant_llm` discovery-complete
- keep it classified as `execution_class = outbound`
- preserve `outbound_network_access = true`
- keep explicit `llm.*` capability opt-in
- make routing enforce that outbound participants are only used when task policy allows them

### 3. Preserve coordinator-managed federation

The coordinator should remain the central control point for v0.2.

This keeps:

- governance centralized at the protocol level
- explainability easier to preserve
- routing easier to understand
- trust boundaries easier to reason about

## Recommended Use Case

The strongest near-term use case for v0.2 is **federated incident investigation**.

Example question:

> Why did the payment system fail on April 7?

Recommended domain participants:

- runbook/docs agent
- observability agent
- ops knowledge agent

Later optional participants:

- outbound vendor-status participant
- outbound LLM analysis participant

Why this use case is strong:

- naturally spans multiple domains
- needs evidence
- benefits from governed local execution
- makes the protocol value obvious

## Milestone Plan

### Milestone 1: v0.1.1 cleanup

Scope:

- doc consistency across README/spec/trust docs
- keep `participant_llm` classified and discovery-complete
- keep the current alpha story coherent

Deliverables:

- aligned participant counts/descriptions in docs
- consistent trust language
- discovery behavior clarified

### Milestone 2: v0.2.0-alpha routing foundation

Scope:

- routing constraints on `fap.task.create`
- coordinator filtering by discovery metadata
- initial execution-class semantics

Deliverables:

- coordinator uses profile/status in routing
- outbound participants only included when allowed
- task creation carries routing intent

### Milestone 3: v0.2.1-alpha richer execution transparency

Scope:

- `fap.task.progress`
- richer evidence typing
- better final-result explainability

Deliverables:

- participants can report intermediate progress
- aggregate result can distinguish evidence types

### Milestone 4: v0.2.2-alpha real demo

Scope:

- one high-quality incident investigation demo
- participant-local tools/MCP inside domain agents
- FAP between them

Deliverables:

- a concrete end-to-end story that shows the protocol matters

## Risks

### 1. Over-expanding into generic MAS claims

If v0.2 tries to become a general "multi-agent everything" platform, it will blur the project's
strongest identity.

### 2. Treating discovery as only descriptive

If profile/status exist but routing ignores them, discovery will feel ornamental.

### 3. Keeping `participant_llm` semantically ambiguous

If outbound participants are not modeled distinctly, they weaken the coherence of the local
execution / governed export story.

## Exit Criteria For v0.2

v0.2 should be considered successful if:

- participant discovery is operational in coordinator routing
- task creation can express routing constraints
- participant classes are explicit
- progress and richer evidence are on the protocol surface
- at least one real demo shows domain-agent style federation clearly

## Final Recommendation

The correct v0.2 path is not more hype. It is more clarity.

FAP should keep strengthening the thing that is already distinctive:

- governed federation across domain boundaries
- explicit message semantics
- explainable aggregated results

If v0.2 does that while making participants more self-describing and more useful as domain
runtimes, the protocol will become much stronger.

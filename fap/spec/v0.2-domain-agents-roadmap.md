# v0.2 Domain Agents Roadmap

This document describes the next protocol direction after the current `v0.1` alpha runtime.

The core idea is:

- **FAP** governs communication **between** domain agents across trust boundaries
- **MCP** stays **inside** a participant as the local tool/runtime interface

In other words, a participant should increasingly look like a domain agent runtime rather than
only a file-backed connector.

## Why This Direction

The current alpha already proves a useful base:

- participants execute locally
- policy is applied before export
- aggregation is explicit and participant-originated
- final answers can carry source-level evidence

The next step is to make those participants more routable, more self-describing, and more useful
as concrete building blocks for real multi-agent systems.

## First Concrete Step

The first useful protocol enhancement is participant discovery metadata:

- `fap.participant.profile`
- `fap.participant.status`

These messages let the coordinator discover:

- what a participant can do
- how it executes
- what privacy/cost/latency posture it has
- whether it is healthy and accepting work

That is the foundation for smarter routing without changing the core governance model.

## Recommended Real Use Case

The strongest near-term demo/use case is **federated incident investigation**.

One example question:

> Why did the payment system fail on April 7?

Domain agents:

- **runbook/docs agent**
  Local MCP tools: filesystem, runbooks, architecture notes, integration docs
- **observability agent**
  Local MCP tools: logs, traces, metrics, alert history
- **ops knowledge agent**
  Local MCP tools: KB, incident notes, postmortems, troubleshooting records

Coordinator behavior:

- route the task to the right participants
- preserve task/run/trace correlation
- collect governed outputs and evidence refs
- aggregate into one explainable result

## Protocol Direction

Recommended follow-on work after profile/status:

1. routing constraints on task creation
   - required capabilities
   - privacy requirement
   - latency budget
   - cost budget
   - allow outbound model calls
2. task progress updates
   - `fap.task.progress`
3. richer evidence typing
   - document evidence
   - log evidence
   - tool output
   - model-generated output
4. coordinator-mediated follow-up turns
5. transport bindings after the semantics are stable
   - HTTP remains first
   - MQTT/NATS/etc. can come later

## Scope Guidance

Recommended now:

- domain agents as FAP participants
- MCP inside each participant
- coordinator-managed routing
- canonical governed aggregation

Recommended later:

- participant-to-participant handoff
- broader A2A patterns
- peer-to-peer topologies

FAP is strongest when it stays focused on governed federation across domain boundaries. That is the
core identity to preserve while the runtime becomes more agentic.

# FAP `v0.1.0-alpha`

## Summary

`v0.1.0-alpha` is the first public alpha release of the Federated Agent Protocol (FAP).

This release should be read as:

- a protocol alpha
- a reference runtime
- a developer preview
- not yet production stable

FAP coordinates governed work across private domains without centralizing raw private data. Participants execute locally against their own sources, export only policy-governed outputs as canonical protocol messages, and contribute evidence-linked aggregation inputs to a coordinator.

## What Is Included

### Protocol And Runtime

- typed protocol core in `fap_core`
- canonical envelope and version-aware dispatch for protocol `FAP` version `0.1`
- DB-first coordinator runtime
- durable `protocol_events` and `run_snapshots`
- trusted participant identity consistency checks during dispatch

### Reference Participants

- `participant_docs`
- `participant_kb`
- `participant_logs`

All three participants now use local file-backed connectors in the reference runtime.

### Governed Execution And Aggregation

- participant evaluation with canonical `fap.task.accept` / `fap.task.reject`
- governed execution with canonical:
  - `fap.task.complete`
  - `fap.policy.attest`
  - participant-originated `fap.aggregate.submit`
- coordinator `summary_merge` aggregation with canonical `fap.aggregate.result`
- source-level evidence refs carried through execution and aggregation payloads

### Integration Surfaces

- `/ask` as a thin user-facing wrapper over orchestration
- `fap_client` as the main Python integration layer
- `fap_mcp` as the MCP bridge

## Package Overview

| Package | Role |
| --- | --- |
| `fap_core` | Protocol SDK and canonical message/runtime types |
| `fap_client` | Thin Python client over the coordinator |
| `fap_mcp` | MCP wrapper over `fap_client` |

Coordinator and participants are shipped as the reference runtime apps in the same monorepo.
For this alpha, the monorepo currently installs as a single Python distribution named `fap`
that exposes these import packages together.

## Important Alpha Limitations

Not yet implemented:

- cryptographic authentication or message signing
- replay engine / full event-sourced rebuild workflow
- richer aggregation modes beyond `summary_merge`
- production-grade deployment guidance
- broader compatibility/version policy beyond `0.1`
- production-real connectors beyond the current local-file-backed reference participants

Current trust posture:

- explicit trusted participant registry
- coordinator-side identity consistency validation
- no cryptographic proof of identity yet

## Next Phase Roadmap

Suggested priorities after `v0.1.0-alpha`:

- replay/recovery over durable protocol events
- stronger state-machine and idempotency documentation
- improved evidence/citation presentation
- stronger trust/auth model
- conformance fixtures for independent implementations

## Release Positioning

This release is strong enough to support:

- demos
- protocol review
- internal experimentation
- early agent integration

It should not yet be positioned as:

- production stable
- security complete
- interoperability final
- enterprise-ready
